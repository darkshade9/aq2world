First off, custom sounds should be in the Baseq2\Sounds\ directory.

There are several ways to get it to work. Make the Target_Speaker and Give it an Attenuation of 3 (Ambient), and for Spawnflags, give it 4 (Reliable). Now this is where u decide wat to do. Make a brush into the Trigger_Once Entity, Triger_Multiple, or Trigger_Always, depending on your situation and target it to the target speaker. This should work, but if not, let me know.

posted 02-11-99 04:30 PM CT (US)          
Actually, placing custom sounds in a custom directory is better (for file management purposes) that placing them in baseq2\sounds. Otherwise you lose track of what sounds are associated with what map (yours or others). In the example given, JazZ has my.wav in baseq2\sounds\mysound.

JazZ - there's nothing in a .wav file that would cause it to loop, so it must be something to do with how you are triggering it. I'd guess you're triggering it with a trigger_multiple that has a short (or no) wait, and standing in the trigger_multiple cussing at the target_speaker that continues to play, not realizing that you're the one causing that to happen (yeah, been there ). If you really only want the sound to play once, use a trigger_once, a trigger_multiple with wait=-1, func_button with wait=-1, etc.

...or if this isn't the problem speak up and we'll figure out something else.

posted 02-12-99 02:34 PM CT (US)          
JazZ - Well, you learn something new every day (if you're lucky ). Thanks for sending the map and .wavs - this was very interesting. I was wrong about there not being anything in a .wav file that could cause this behavior. I don't have Sound Forge (which you used), but I do have a shareware package called GoldWave. I opened up your old.wav and saw something curious that I'm still not sure of the purpose for: your old.wav has a "cue point" (I assume this is a GoldWave tech-term) at the start of the .wav (00:00:00), and new.wav didn't have this. I removed the cue point from old.wav and it then played only once, as expected. I then added a cue point at the start of new.wav, and it now repeated as old.wav previously did. So there ya go.

As an aside - this is useful information for custom sounds that you want to loop indefinitely and be heard across an entire level. The solution for "normal" .wavs is to use a func_timer set to the duration of the .wav. With custom sounds, though, you can add a cue point at the start of the .wav and voila - no func_timer required.

Woo!! I get to show some "elite" knowledge here

I have had quite a bit of custom sound experience with personal maps and Ranger Movies.

Actually, you can record a sound file (.wav) and have it loop in a map and it has NOTHING to do with the map, the entity spawnflags,what directory you use, etc..

It has to do with the sound file and what or how it was recorded and edited.

For example:

I use Soundforge to record. Soundforge places markers for editing in the .wav file for such things as recording sessions....like voice recordings for example. These markers are saved in the .wav file format. Other programs may do something similar. If left in, these markers will cause the infamous "custom sound looping problem". With Soundforge, you can delete these editing markers...and thus the sound will not loop. I am assuming that the Quake 2 sound code does not know how to read this "marker" data and this causes the error which manifests itself as looping.

Somehow when Jazz re-recorded it, or cut and pasted his sound data, such markers or their equivalents were eliminated..and thus the seemingly "magic" event of his sounds working properly for no apparent reason.

This is just one of those odd things when doing custom sounds for Quake 2, the other is your sound volume....you have to really increase the volume sometimes of your .wav file.

Hope this helps or clears stuff up, I have very primative knowledge of the .wav file format so that was a layman's explanation above I am a level designer, not a sound technician or coder!

Good luck, hope this helps
