posted 02-15-99 11:23 PM CT (US)          
I believe the readme.txt file that comes with SinEd clearly explains how to set up a Windows desktop shortcut so you don't HAVE to start it with runsined.bat.

One more time for the world, the shortcut has to be configured:

c:\tools\sin\bin\sined.exe ..\base\scripts\sample.sed

The project file is the sample.sed file here. When you start SinEd, you have to specify a project file so it knows where (what path) to find textures, models, entity definition files, surface files, etc...
It is simply a kind of configuration file, nothing more complicated that that.

So unless you have been creating a new .sed (project file) file (which would be totally unneccessary and asking for trouble anyway), you never have to do a "file" "load project". In fact, just don't use this unless you REALLY know what you're doing OK?

So here's the picture (once more for the world):

You start SinEd (preferably from the Windows desktop shortcut). After you're done with GL thing, this automatically opens an empty map named : unnamed.map

Never mind that, that's normal.

Then, go to the "textures" menu and select any texture folder (bank, aqueduct, whatever). This is what loads textures from each texture sub-folder into SinED.

Then ONLY, go to the "file" menu and do an "open" , "yourmap.map" (don't touch that "load project" thing!!).

Then your map with all the textures you alredy used in it will be loaded but now you should be able to load any texture sub-folder you need afterwards.

I hope this clears it up...

Ok one last try.

When you unpacked sintools_fill.zip, did you use Winzip? Did you extract with the "use folder info" option checked?

I once saw someone who unzippped the whole contents of sintools_full.zip into the c:\tools folder without letting Winzip create all the necessary subfolders under "tools" and all the files ended up in the "tools" folder itself with no sub-folders created. Could this be the case here?

If not, do you have VisualBasic 5 or 6 on your machine? Did you use it to create new OCX's od DLL's that ended up in your windows/system folder?

I have a friend on which's machine SinEd refused to work completely because of this. Shortly afterwards, his OS crashed completely and he re-formatted the HD, re-installed Win95 and then SinEd right away and it worked perfectly.

Visual Basic was causing the problem IMO. Or Matrox's MOC utility IMO. There was something on his machine that SinEd didn't like.

Hope this can help solve it.
