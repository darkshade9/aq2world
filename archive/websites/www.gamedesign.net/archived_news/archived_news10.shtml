 Worldcraft 2.2 Info (3:41) Updated by [ weasl ] 

Saw over at the Half-life ERC some cool info regarding changes in Worldcraft 2.2. Here's the meat.
    The next update (Worldcraft 2.2) will included a number of tasty new things (these were mentioned specifically for the next update).
    - "go to brush #" and "go to map coords" features to make finding problems reported by the compile tools easier
    - true texture locking that works with rotation (!!)

    "We're trying to fix all the things that make level editing unnecessarily time-consuming. Fiddling with texture coordinates and geometry that breaks up doesn't make anyone more productive."

    "We've talked about a sphere tool and we're looking into curves as well. We don't see the need to do dynamic tesselation in the engine at this time."

    "For now we're focusing on allowing more complex geometry without it breaking up when brought into the engine. We want to remove the unit grid limitations in Worldcraft, for example."

    "Worldcraft will definitely be our level design tool for the forseeable future. And that's a good 2 years or so."

    [regarding hierarchical defined VisGroups] "I'm very definitely planning on that."

    "Yes" to game-profile specific texture settings.

    "No plans at the moment for plug-in support in Worldcraft. Maybe if we get really good feedback on what people would like to see it would become a higher priority."
Also 2.2 should bring the return of Quake/Quake2 support to Worldcraft as well.

 New Sin .cpp file (23:37) Updated by [ Bushboy ] 

The Node has released a totally revised and updated Sin entity .cpp file, here's the lowdown from EutecTic:-

Whenever I get my hands on a new game, I can't help but to wonder: how does this entity work?... how come I set this key and nothing happens?... Or you'll set a key that's not listed in the info and find that it works after all and think: geez! How many more of those that weren't listed here work with this entity?... And how many more entities in the set have missing information like this?... A lot probably?...

Well call me obsessed if you want but not knowing for sure drives me nuts. So I decided to rectify this situation and sort it all out by testing those suckers one by one and trying just about everything I could think of and this is the result. A brand new shiny, accurate entity definition file plus 7 updated .def files for good measure :)

To give you an idea, the .cpp file went from about 1100 lines to 1575 lines (word wrap on). The file size went from 53K to 100K. And since it's a text file, I even give you permission to read it in your spare time between making mods ;)

 QERadiant, Java, and Future of QERadiant (17:05) Updated by [ Fishman ]

    Machines are getting faster and cheaper so I think in large, this problem goes away (at least with the current level of level design complexity).     I am starting the process of a cross platform editor. It will be largely Radiant based but will be written in C++ with a System layer that is written for each platform. Since GL performs 90% of the ui, standard libraries can be used across systems. It will be an interesting exercise.

 New features is 224 & UT (11:25) Updated by [ Wolf ] 

I saw thins on Adrenaline Vault

Another major enhancement to be included in 224 is continuous level of detail, which varies polygon counts on objects such as player characters and power-ups relative to their distance from the camera, or player's perspective. In Unreal Tournament, this will enable the engine to place more characters and objects on the screen and still maintain a consistent and fluid framerate. Rein said this is an important feature not just for Unreal and Unreal Tournament but also for engine licensees since it also opens the engine to new game types, such as real-time strategy titles, that were not previously obvious choices for the engine.

Level of Detail is a great enhancement and should speed up the frame rates. I still say this patch is gonna rock!
