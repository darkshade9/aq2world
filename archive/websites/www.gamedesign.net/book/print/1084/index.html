    dod_object and dod_object_goal combinations can be used for both primary and secondary objectives.

    In DoD v1.X a dod_object_goal is no longer required to make a simple wall that can be demolished with TnT (dod_object) as all the settings required such as "objective group", "time delay", "trigger when tnt is set" can all be made directly on the func_breakable representing the wall.     The difference between a primary and a secondary objective is the final target entity for the completion of the objective. For a primary objective the target entity will be a dod_score_ent (possibly via another entity such as a multi_manager) which will end the round when the objective is completed. For a secondary objective the target entity can be virtually anything depending on the desired result of completion.

    dod_object_goal has no team specific function, if you wish to make team specific object objectives then the objects / goals should be separated into two objective groups (one for each team), and the team specific options set on the dod_object.

    It is possible to have a mixture of team specific objects and non team specific objects in a single group, it is also possible to have some of the objects in a group specific to one team and others specific to the other team. Spawn Advance

    Mapping Day of Defeat - by Nick Robins

    Advancing Spawns

    DoD spawns do not actually move, the SoHL "movewith" feature does not work with the player start entitys. The spawns are "moved" by turning one set off and another set on, creating the illusion of movement.

    Spawn activity is controlled by using masters. There are two mastering methods available to a mapper for controlling whether a spawn point is active at a given point.

    Using a Half-Life master

    A single half-life master can control two sets of spawns for each team by using the "invert master" SoHL function (more spawn stages can be set up but the entity set-up will be quite complex). This method also makes setting up spawn protection (using the modified trigger_hurt or trigger_teleport entitys) relativly simple as these can be set up to use the same master as the player spawn points.

    Using a DoD master

    DoD's "spawn cp master" function using a single "master flag" can control three sets of spawns for each team. This method is simpler to set up (for a basic system there are no extra entitys required other than the extra spawns), but spawn protection can be harder to set up.

    It is possible to have more than three spawn stages using either method but this would require an entity set-up that could get quite complex. In addition to problems with complexity, each spawn set (for a single team) should consist of 20 info_player_[team]'s, thus for three spawn stages per side would mean a total of 120 (20 spawns x3 stages x2 teams) entitys are required for the player starts alone. This could cause the entity limit to be hit quite quickly.

Protect Spawns

    Mapping Day of Defeat - by Nick Robins

    Spawn Protection

    There are several methods and entitys that can be used to protect spawn areas in DoD. These fall into two broad categories.

    • Physical - players cannot enter the spawn area from the outside without help.
    • Punishment - punitive action is applied to players in a forbidden area.

    Usually a combination of methods will be used. A normal set up is to attempt to exclude players physically and to have punishment as a back up for those people who succeed in geting past.

    Physical protection

    Ramps

    This is the simplest method and requires no entitys, simply arrange your spawn exits with ramps shaped such that a player can run up and over the ramp from the spawn side but not from the map side. It may still be possible to access the spawn area via player boosting. This method also requires no extra work for maps where the teams change ends.

    Just a note from Andy..
    Too many DoD maps suffer from spawn areas that are easily attacked and the spawn area covered by direct fire of the opposite team. I have always thought that the best spawn area, was one that allowed spawning players the ability to exit several areas.

    Team only doors

    func_door and func_door_rotating both have options to make the door team specific. This is relativly easy to set up. The team specific nature of the doors only prevents enemy players from opening the doors, it does not prevent enemy players from running in when the door is open (this can sometimes be solved by using double doors (similar to an airlock).

    Switching doors

    In order to protect a spawn that can be used by both sides (eg dod_schwetz) with team specific doors, it will be nessasary to set up the doors to only open when triggered, usually using a trigger_multiple. If the trigger is inside then the spawn will be mostly secure. Two team specific trigger_multiple(s) can also be used (the one that is not desired can be disabled using a master).

    Pushing players

    A trigger_push placed in the spawn exit can prevent anybody from entering the spawn area from the map.

    Punishment

    Hurting offenders

    The entity trigger_hurt in DoD has two flags added ("don't hurt allies" and "don't hurt axis") enabling mappers to world kill (quickly or slowly) any enemy player who enters a spawn area. This entity can also be enabled and disabled using masters, allowing the spawn area to be used by either team if required.

    Teleporting offenders away

    trigger_teleport has two flags added ("no allies" and "no axis") to make it team specific, enabling the rapid removal of a player from a spawn area (usually to a prison or somewhere very dangerous). Trigger_teleport can also have a master.

    Removing an offenders weapons

    Player_weaponstrip is a standard half-life entity which removes all of the triggering players weapons. Since players can pick up weapons in DoD 1.X this may not be a very effective method of punishment on it's own.

    Reducing an offenders score

    Game_score is a standard half-life entity which alters the triggering players score. By specifiying a negative number in the "points to add" field then the players score will be reduced. there is also an "allow negative" flag.

    Blowing offenders away

    The various func_tank entitys have been modified to allow them to be friendly to either allies or axis, this allows you to make "autocannons" that will attack any "enemy" that they see.

    Punishment notes

    trigger_hurt and trigger_teleport will affect a player as soon as they enter the trigger volume.

    player_weaponstrip and game_score require the player to trigger them. This is best acomplished using a team specific trigger_multiple if the map requires the teams to change ends then multiple triggers should be used with masters to control which trigger is active.

Using Masters

    Mapping Day of Defeat - by Nick Robins

    Masters

    The Half-Life master system is a method of enabling and disabling entities based on the status (on or off) of another entity. Generally this will mean that if the master is in state off then the entity will be disabled (but see below on using Spirit of Half Life prefixes). An entity which is disabled will not fire if triggered but will complete a sequence if it was triggered before its master was disabled.

    Master entities

    A master in half life will usually be a multisource, in DoD multisources can be used and in some cases will be the only way of achieving the desired effect.

    Multisource

    As its name implies a multisource can have a number of inputs. These are tested using logical AND, that is the multisource will only be active if all of its inputs are active. in this respect the multisource has been made obsolete by the SoHL entity multi_watcher which allows a choice of logic tests to be applied.

    Note problems with multisource and multi_watcher: Neither of these entities appears to unset (for re-triggering) at the start of a new round. This is possibly due to them being implemented for a single player game.

    A multisource can also have its state set by a global variable set by an env_global, the important point here is that a global variable does not reset when a new round is started. This means that if you wish to have different spawn locations or mission objectives in different rounds you must use an env_global/multisource combination somewhere in the control sequence.

    Note: The global variable does not reset when a new map loads, as long as the server remains running the global variable will remain. If the "set initial state " flag is not set then the global variable can affect the map when (or if) the server rotates back to the map where it was set.

    Env_state

    Spirit of Half Life

    Env_state is an entity included in DoD from SoHL. It is designed to be used as a master. At the simplest level the job of an env_state is to be either off or on and to toggle between these two states, although there are additional features available, eg an env_state can fire another entity on state changes.

    A useful trick with an env_state is explicitly setting its state. By creating a trigger_relay (or other entity) targeting the env_state, with "trigger to send" set correctly ("on" or "off") you can create an entity which will force the env_state into the state you desire (if the env_state is already in the desired state, the trigger will still fire the env_state but will have no effect). This can also be achieved by using a prefix

    Other entitys

    Spirit of Half Life

    Using Spirit ANY entity can be a master.

    The majority of the time either a multisource or an env_state will be the preferred option as a master. However in some circumstances this SoHL feature can allow a significant reduction in the number of entities required for an effect and/or be much more convenient.

    SoHL prefixes

     Spirit of Half Life

    • Prefixing the name of a target entity with + produces a 'turn on' trigger, eg +light1 will only set light1 into state "on".
    • Prefixing the name of a target entity with - produces a 'turn off' trigger, eg -light1 will only set light1 into state "off".

    Note: I have had some problems using this feature; it does not always seem to work

    If the entity is already in the required state then the trigger request will still go through but will have no effect.

    • Using tilde ~ as a prefix to an entity named as a master will invert the master relationship, the entity will only be active when the master is inactive. For example an entity with a master named as follows:

    Master: ~group_master

    Will only trigger if 'group_master' is in state "off".

    This is a imensely useful feature. Often it is the case that you want to have two systems, only one of which can be active at a time, previously this would require two masters and extra entities to keep them in sync and opposed in state, now just the one master entity is required and much simpler control gear which just has to toggle the master on and off.

    Masters in DoD

     DoD also has two areas of interest when using masters.

    1. Using a master with a dod_control_point_master

    The master named for a dod_control_point_master applies to all the dod_control_point(s) in the objective group that the dod_control_point_master acts for.

    If the master of the dod_control_point_master disables the entity then the whole objective group is disabled (and will have no visual presence in the game).

    Note: A dod_control_point does not have a 'master' field, using the dod_control_point_master is the only way to disable this entity. Also this effect does not work with dod_object and dod_object_goal

    1. Using a dod_control_point as a master

    This is only applicable to DoD player starts (info_player_allies and info_player_axis) and allows you to use a dod_control_point as a tri-state master (spawncpmaster or "Master Control Point"). Enter the name of the dod_control_point you wish to use as a master in the "Master Control Point" field. You can then select whether or not the spawn should be active for each of the three possible flag states.

    Note: The name required is the targetname of the dod_control_point NOT the "control point name".

Placing Models

    Mapping Day of Defeat - by Nick Robins

    Placing Models

    There are several entities that can be employed to place a model in your map.

    • dod_object
    • dod_control_point
    • env_model - from Spirit of Half Life (new in beta 3.0)

    Each entity has advantages and disadvantages.

    dod_object

    Pros

    • Simple
    • Automatically drops to ground

    Cons

    • Requires clipping to prevent players from picking up object
    • Automatically drops to ground
    • Default animation only

    dod_object is the simplest option for placing a model. A dod_object will always drop to the ground automatically which means that vertical placement need not always be 100% accurate but the object may require a "platform" to sit on (this can often be accomplished using clipping brushes).

    If a dod_object can be reached by players it must be clipped or a player can pick up the object which could look rather odd.

    dod_control_point

    Pros

    • Supports multiple models
    • Configurable body
    • Easily toggled to change model
    • Unaffected by gravity
    • Properties can prevent players from toggling the model

    Cons

    • Can be difficult to place correctly
    • Default animation only

    A dod_control_point can have up to three models specified (one for each of the three states the flag can be in). dod_point_relay(s) can be used to toggle the flag between states (which will change the displayed model).

    A dod_control_point can be hidden using either an env_render or by disabling the master of the relevant dod_control_point_master

    dod_control_point is not affected by gravity so to ensure the model is not levitating or submerged you may need to tweak the precise placement of the entity.

    Clipping is only required to prevent players from running through the model as by setting "can allies cap" and "can axis cap" to 'NO' players will be prevented from toggling the point (and therefore the model).

    env_model

    Pros

    • Multiple animations
    • Easily toggled to change animation
    • Can be affected by gravity (mapper's choice)
    • Configurable skin
    • Configurable body

    Cons

    • None really

    env_model is an entity specifically designed for placing models and has options for configuring animations, skin and body.

    env_model can be either "on" or "off", this does not mean present or not present. The on and off states are used to control the animation, env_model will play one animation while "on" and a second while "off".

    Clipping is only required to prevent players from running through the model.

    Notes on positioning models.

      
    click to enlarge

    If the entity you are using to display the model is unaffected by gravity (dod_control_point and env_state) then you need to make sure the model is not "levitating".

    Load the model in HL model viewer and set the option to display "bones", you will see a small red dot (see left picture). This is the models origin and will be at the "point" where the entity is. Usually (with a standard FGD) this is the center of the visible entity box (the "box" you see in hammer is purely a visual aid).

    With most models this means you just make sure that the brush face forming the ground bisects the entity box (see right picture).

    Clipping models

    Models require clipping to prevent players running through them (and in the case of a dod_object to prevent the player picking the object up), if the model is located in an inaccessible place and is not required to be bullet proof then clipping can be omitted.

    Clipping with clip brushes

    A clip brush is a brush completely covered with the "clip" texture (found in valve/halflife.wad). This will prevent the player from running through the model but weapons (both bullets and grenades) will penetrate.

    Special Clipping Textures

    This is currently the best method of model clipping available. It involes the use of a func_wall.

    Special textures are available in hlbasics.wad that make the use of func_buttons and ambient_generic's pretty much redundant.

    •   cr_mathvymetal
    •   cr_matleaf
    •   cr_matmetal
    •   r_matsand
    •   cr_matwood

    This is not an exhaustive list as all textures listed in the materials.txt (found in your sound folder) can be used to produce a suitable clip type brush. The materials .txt also gives you the effect the texture will provide when used.

    Build your brush textured with any of the textures above, to cover the object you want clipped. Assign the brush to func_wall and set the Render Mode to 'Texture' and the FX Amount to '0'.
    What this does is assign the brush a sound defined in the 'materials.txt' and also give the brush clipping properties aligned with the material type. If you chose 'cr_matmetal' then when that brush is shot it will produce a spark and a richochet sound. The brush textured with 'cr_matmetal' will not be penertrated by bullets.

    Clipping with func_button entities

    To make a model bullet proof, it must be covered with a brush entity. Place a brush or brushes covered with a suitable texture over the model. In DoD v1.0 hit effects (bullet puffs, impact sounds etc) are controlled by the name of the texture being hit (even if rendered invisible) so for example to clip a tank model you would want to use a "metal" texture.
    Select the brush and go to tools - tie to entity (ctrl-t). Change the entity to func_button and alter the properties as follows.

    This means that any bullet hitting the func_button will be absorbed to trigger the button (this is one of the effects of the "health" value) effectively making the button bullet-proof. if you want a ricochet noise to sound when the model is shot, then place an ambient_generic in a suitable location and set the func_button's target to trigger it.

    Notes:

    If the model is going to change shape significantly, either from toggling a dod_control_point or by applying env_render(s) to multiple dod_object or env_model entities then func_wall_toggle may be required to ensure that the clipping always matches the shape of the model.

    Getting the clip(s) to coincide with the model is mostly down to trial and error as Hammer does not display the model (NB, Hammer 3.5 beta can display models in the 3d veiw). the number of brushes to use to clip a model is also tricky. The more you use the closer the model will be clipped, however this will make your clip node count higher.

    Class & Weapon Limits

    Class limits are usually defined in a file named <mapname>.cfg in the servers DoD folder. This will be automatically executed by the server when the map is loaded (note: the map cfg file is exec'd beforethe config file specified in the server cvar mapchangecfgfile). The map config file can be used to adjust any server cvar, but is most often used to provide map specific class limits. The file is also exec'd (if it exists) on the clients so it can be used to set client side behavior as well, such as the fog parameters set in the dod_dog1 config file.

    While you can set most cvars for your map on both the client and server using this file, some discretion is advised. Many clients and nearly all servers have quite finely tuned configs, if you change these you could make yourself unpopular.

    Setting a limit of -1 makes the weapon unlimited. Setting a limit of 0 (zero) disables the class from being selected, with DoD v1.2 you can remove classes from the selection screen altogether.

    You cannot have both the US and the UK in a map at the same time but it dosn't matter if a map's config file has cvars for limiting the unused team, so if you want to create a generic config and copy it to all the various map configs it will work fine. However if you are using the cvars to remove classes from the selection menu, you should make sure that you have the team selection correct.

    This information is correct up to DoD Version 1.0b (WON and STEAM version).

    STEAM

    STEAM clients can have custom content installed very easily. The directory structure to use is identical to the WON version, the only difference is the location of the "dod" folder.

    The dod folder in a default STEAM install is:

    C:\Program Files\Steam\SteamApps\<email address>\day of defeat\dod

    <email address> is the email address used to create the steam account.

    Custom content should be unpacked to this folder the same as the WON version's C:\SIERRA\Half-Life\dod.

    Note: by default the steam "dod" folder dosn't contain all the subfolders you would find in a WON install of DoD. This is not an issue since they will be created as required by the installer / archive tool.

     

    Server Compatability issues

    ← This guy is your friend.

    More than half of the dedicated DoD servers (about 60% last time I checked) out there are classified as "Linux" servers (note that these are not all actually running GNU/Linux, many will be running one of the BSD distributions).

    When a server loads a map, it also loads all the required resource files (sound .wav, texture .wad, model .mdl etc) needed for the map. If the server cannot find one of these files, it dies.

    Microsoft® Windows® is not case sensitve with filenames. Unix (whether it be GNU/Linux, *BSD or a comercial UNIX) IS case sensitive with filenames. This means that if a filename in your map has a case error in it, a "Linux" server will be unable to find the file and therefore unable to run the map.

    A classic case (which I have found a fair few custom maps suffer from) is this:

    map requires: dod/models/Tnt.mdl
    actual filename: dod/models/tnt.mdl

    Most servers running custom maps have probably already implemented work-arounds to the more common examples (such as the TnT model mentioned above) but there can always be new ones. DoD v1.0 has re-shuffled most of the resource locations, which will likely remove most of the work-arounds implemented during the DoD beta's by the server operators.

    Pay particular attention to the capitialisation of resource filenames where they are specified in your map. Windows® makes this slightly harder due to the fact that it auto-capitalises filenames when they are displayed (but doesn't actually change the filename itself).

     

    Which files to include

    Chances are your map uses some resources which are not part of the standard fileset of DoD. Since Half-Life can't run the map without the required resources, you need to make sure that all resources your map requires will be present.

    An easy way to find out all the files your map requires is to use RESGEN (from United Admins). When this is run on a BSP file it produces a RES file (plain text) listing all the required files. RES files (which should be located in the maps folder) are also used to tell the server which files a client needs to download from the server (if downloads are enabled) so it's a handy file to include with your map anyway.

    Make sure that you only include those files which are NOT part of a standard DoD install, otherwise your map installation may overwrite peoples customisations (such as regional flags and/or victory music).

    Map specific files

    (None of these files are required but form part of a "complete" distribution)

    dod/maps/<mapname>.txt

    This is the "Mission Instructions" that will be shown to the player when the map loads. Usually also used to credit people who's work is included in the map (such as textures, models, sounds etc).

    dod/<mapname>.cfg

    The config file for the map, usually used to set the class limits

    dod/overviews/<mapname>.bmp
    dod/overviews/<mapname>.txt

    The HLTV overview files (graphic and configuration)

    dod/maps/<mapname>.res

    The resource file for the map.

    Packaging files

    I use the following convention for distribution (these are merely my preferences, not the law):

    • The archive is a ZIP file.

    ZIP files are supported by default in the current generation of Windows®, they also present few problems to Linux. Personally I dislike executable installers (unless they can be opened with an archive tool) due to the fact that it's a pain to then get all the files onto a Linux system.

    • The "root" of the installation is the level above the "dod" folder.

    A ZIP file can (and in this case should) contain the folder information for each file. I prefer to make the archive so that the file locations are stored thus:

    dod/maps/<mapname>.bsp
    dod/maps/<mapname>.txt
    dod/<mapname>.cfg
    etc

    This makes it clear, when opening the archive, which Half-Life Mod the map is intended for, and also allows equally easy installation to Half-Life or either of the "retail" (Counter-Strike and DoD) versions.

    Notes:

    dod_control_point_master and objective groups
    Since beta 2.0 DoD has had the facility to have objective groups which allows much easier control of mission objectives and sub-objectives. The two most important points about groups and the dod_control_point_master are:

    1. Each flag (dod_control_point) objective group must have a dod_control_point_master.
    2. Disabling a groups dod_control_point_master (by switching it's master off) will disable the whole group.

    A flag groups dod_control_point_master dosn't necessarily need to perform any function for the group but the flag group needs it to be there. Normally the only time this is a problem is when using hidden flags to control spawns or events.

    Disabling the group means that none of the groups entitys will trigger and all signs of them (such as models) will be hidden. This allows a map to have several objective systems, which can be switched on and off on the fly. Pre DoD v1.0 this was the only way to disable a dod_control_point as this entity did not have a 'master' field (added DoD v1.0).

    A group consisting of dod_object(s) and dod_object_goal(s) does not require a dod_control_point_master.
