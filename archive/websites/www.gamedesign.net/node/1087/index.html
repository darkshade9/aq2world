Mapping Day of Defeat - by Nick Robins

Advancing Spawns

DoD spawns do not actually move, the SoHL "movewith" feature does not work with the player start entitys. The spawns are "moved" by turning one set off and another set on, creating the illusion of movement.

Spawn activity is controlled by using masters. There are two mastering methods available to a mapper for controlling whether a spawn point is active at a given point.

Using a Half-Life master

A single half-life master can control two sets of spawns for each team by using the "invert master" SoHL function (more spawn stages can be set up but the entity set-up will be quite complex). This method also makes setting up spawn protection (using the modified trigger_hurt or trigger_teleport entitys) relativly simple as these can be set up to use the same master as the player spawn points.

Using a DoD master

DoD's "spawn cp master" function using a single "master flag" can control three sets of spawns for each team. This method is simpler to set up (for a basic system there are no extra entitys required other than the extra spawns), but spawn protection can be harder to set up.

It is possible to have more than three spawn stages using either method but this would require an entity set-up that could get quite complex. In addition to problems with complexity, each spawn set (for a single team) should consist of 20 info_player_[team]'s, thus for three spawn stages per side would mean a total of 120 (20 spawns x3 stages x2 teams) entitys are required for the player starts alone. This could cause the entity limit to be hit quite quickly.

 

Half-Life master

Using an env_state as the master

Entitys used:

  • info_player_allies
  • info_player_axis
  • Env_state
  • Trigger_relay
  • Func_button

See related documents for non DoD entity references

Step one: Add the master

Select the point entity tool (shift-e) and select env_state from the drop down menu in the sidebar on the right. Place the entity somewhere suitable (this entity has no visible presence in game so place it somewhere convenient for you. Press alt+enter to bring up the properties.

Step two: Master controls

Select the point entity tool (shift-e) and select trigger_relay from the drop down menu in the sidebar on the right. Place two of these entitys somewhere suitable (this entity has no visible presence in game so place them somewhere convenient for you, such as either side of the master).

  

Step three: Adding the spawns

Select the point entity tool (shift-e) and select info_player_allies from the drop down menu in the sidebar on the right. Place one of these entitys at one end of your map, place an info_player_axis next to it.

  

Note the tilde ~ in the master field for the info_player_axis, this inverts the master.

Add another info_player_allies and another info_player_axis at the other end of the map. These will need their masters set to the opposite state.

  

Once these four are in place the remaining desired player starts can be placed by copying.

Step four: Adding switches for testing

Normally in a playable map the spawns will be switched or moved by an event ocouring, usually the completion of an objective (major or minor) by one or other team. For testing purposes func_buttons can be used to alter the spawns.

Create two func_buttons, place the brushes and select tools - tie to entity (ctrl-t) change the entity type to func_button

  

Testing moving spawn entity set-ups can be a pain as you have to operate the entitys, then kill yourself in order to make sure you respawn at the correct place, this process may have to be repeated over and over.

Optional

Hiding the master control point

The dod_control_point that is controlling the spawns does not have to be one of those required for a mission objective. The master dod_control_point can be put into a separate objective group, and hidden (both visually and on a players HUD). The state of the dod_control_point can be controlled using dod_point_relay(s).

A hidden master flag group should consist of one dod_control_point_master and two hidden dod_control_point(s). The second dod_control_point should be set to be neutral. Its purpose is to prevent the groups (required) dod_control_point_master from constantly triggering when the master flag is "held" by a team.

It shouldn't be nessasary to set the "can [TEAM] cap" fields, just place the hidden control point(s) well out of the reach of the players.
